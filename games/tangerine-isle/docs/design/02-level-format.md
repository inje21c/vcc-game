# 02 — 레벨 포맷 및 검증기(솔버) 명세: 감귤섬 삼총사

**문서 버전**: 1.0
**상위 문서**: `01-gdd.md` (규칙의 유일한 출처 — 본 문서는 그 규칙의 데이터 표현과 검증 방법만 정의한다)
**저장 위치**: `vcc-game/games/tangerine-isle/docs/design/02-level-format.md`

---

## 0. 설계 원칙 두 가지

**원칙 1 — 규칙 엔진은 하나다.** 게임 런타임과 레벨 솔버는 동일한 규칙 모듈(`src/core/rules.js`)을 공유한다. 규칙이 두 곳에 구현되면 반드시 어긋난다. 솔버가 "풀 수 있다"고 판정한 레벨이 게임에서 안 풀리는 사고는 이 원칙 하나로 원천 차단된다.

**원칙 2 — 레벨은 사람이 읽고 고칠 수 있어야 한다.** 지형은 문자 그리드(한 글자 = 한 타일)로, 오브젝트는 좌표 목록으로 분리한다. 지형은 눈으로 보고, 오브젝트는 속성(폭탄 범위, 문-버튼 연결)을 갖기 때문이다.

## 1. 파일 배치

```
games/tangerine-isle/
├── src/
│   ├── core/rules.js          규칙 엔진 (런타임·솔버 공유)
```

## 2. stages.json 스키마

### 2.1 최상위 구조

```jsonc
{
  "version": 1,
  "tuning": { "G": 10 },              // 귤 1개 = 체력 +G (전역 기본값)
  "stages": [ { /* Stage */ } ]
}
```

### 2.2 Stage 객체

```jsonc
{
  "id": "1-1",
  "chapter": 1,
  "title": "감귤 과수원",              // 표시명 (로컬라이즈 대상)
  "startHp": 60,                      // S₀
  "targetAlpha": 2.0,                 // 설계 목표 여유율 — 솔버 산출값과 대조
  "roomGrid": [2, 1],                 // 룸 배열 [가로, 세로]
  "startRoom": [0, 0],
  "startPos": [7, 12],                // 룸 내 타일 좌표 [x, y]
  "rooms": [ { /* Room */ } ]
}
```

### 2.3 Room 객체 — 지형층 + 오브젝트층

```jsonc
{
  "coord": [0, 0],                    // roomGrid 상의 위치
  "terrain": [                        // 15행 × 15자 문자열
    "TTTTTTT..TTTTTT",                // 상단 '..' = 위 룸으로의 연결부
    "T.............T",
    "T....TT...T...T",
    "T....P........T",
    "T....P....d...T",
    "T.....~~~~....T",
    "T.....~~~~....T",
    "T.............T",
    "T....TTT......T",
    "T.............T",
    "T......D......T",
    "T.............T",
    "T.............T",
    "T.............T",
    "TTTTTTTTTTTTTTT"
  ],
  "objects": [
    { "type": "rock",      "pos": [5, 8] },
    { "type": "rock",      "pos": [6, 8] },
    { "type": "button",    "pos": [3, 3],  "group": "g1" },
    { "type": "button",    "pos": [4, 3],  "group": "g1" },
    { "type": "bomb",      "pos": [9, 4],  "range": 2 },
    { "type": "bomb",      "pos": [12, 4], "range": 3 },
    { "type": "tangerine", "pos": [2, 6] },
    { "type": "key",       "pos": [13, 2] },
    { "type": "chest",     "pos": [7, 11] },
    { "type": "stake",     "pos": [1, 1] }               // 서사 장식, 규칙 무관
  ],
  "doorGroups": { "D:10,6": "g1" }    // 문 좌표 → 버튼 그룹 매핑
}
```

### 2.4 지형 문자 코드

| 문자 | 타일 | GDD 참조 |
|---|---|---|
| `.` | 바닥 `t_ground` | §5.1 |
| `T` | 나무 `t_tree` — 토끼 관통·폭탄 파괴 | §5.1 |
| `~` | 물 `t_water` — 거북이만 | §5.1 |
| `P` | 구덩이 `t_pit` — 돌로 메움 | §5.1 |
| `d` | 나무 비밀문 `t_door_wood` — 버튼 or 폭탄 | §5.1 |
| `D` | 돌 비밀문 `t_door_stone` — 버튼만 | §5.1 |

룸 연결: 룸 가장자리의 `.`가 인접 룸의 같은 위치 `.`와 맞닿으면 통행로다. 별도 선언 없이 지형이 곧 연결 정보다 (검증기가 양쪽 정합성을 확인한다).

## 3. 솔버 명세

### 3.1 상태 정의

```
State = (활성캐릭터, 위치(룸,x,y), 돌배치집합, 잔여폭탄집합,
         제거된지형집합(나무·문), 메워진구덩이집합,
         수집귤집합, 열쇠보유여부)
```

체력은 상태에 넣지 않는다 — 아래 2단계 분리가 탐색 폭발을 막는 핵심 설계다.

### 3.2 2단계 검증

**1단계 — 공간 논리 (체력 무시):** 상태 그래프 BFS로 "상자 개방" 도달 여부와 최단 스텝 `MinSteps`를 구한다. 이동·캐릭터 전환·돌 밀기·기폭이 간선이다. 전환은 스텝 0 (GDD §4).

**2단계 — 체력 수지:** 1단계의 해 경로들에 대해 `HP(t) = S₀ + 수집귤×G − 누적스텝 > 0`이 전 구간에서 유지되는 경로가 존재하는지 확인한다. 귤을 추가로 주우러 가는 우회가 필요한 경우 그 우회 포함 경로로 재탐색한다.

### 3.3 출력

```
$ node tools/solver/solve.js 1-1
✔ SOLVABLE   MinSteps: 84
  α (computed): 1.9   target: 2.0   → ⚠ 목표보다 빡빡함 (귤 +1 또는 S₀ +8 권장)
  필수 캐릭터 전환: cat→turtle(물 구간)→cat, 최소 2회
  폭탄 순서 제약: bomb[12,4](3) 를 bomb[9,4](2) 보다 먼저 기폭하면 데드락
  데드락 진입 가능 상태 수: 3 (런타임 감지 대상)
```

### 3.4 정적 검증 규칙 (validate-all.js)

| # | 규칙 |
|---|---|
| V1 | 스테이지당 key=1, chest=1. 열쇠 없이 상자 도달 경로가 있어도 개방 불가 처리 확인 |
| V2 | 모든 문의 버튼 그룹이 존재하고, 그룹 버튼 수 ≤ 도달 가능한 돌 수 |
| V3 | 룸 외곽 폐쇄성 — 연결부를 제외한 가장자리는 차단 타일 |
| V4 | 룸 연결부 양측 정합 (한쪽만 뚫린 통로 금지) |
| V5 | 솔버 SOLVABLE + `|α계산 − α목표| ≤ 0.2` |
| V6 | 시작 위치에서 첫 귤까지의 스텝 < S₀ (초반 즉사 방지) |
| V7 | stake 등 서사 오브젝트가 통행·규칙에 간섭하지 않음 |

### 3.5 복잡도 통제

돌 배치 조합이 상태 공간의 지배 항이다. 통제 수단: ① 룸당 돌 ≤ 6개 (레벨 설계 제약으로 명문화) ② 상태 해시 + 방문 집합 ③ 밀어도 의미 없는 돌 위치(벽 구석 데드스퀘어) 가지치기 — 소코반 솔버의 표준 기법. 17스테이지 규모에서는 이 세 가지로 충분하며, 초과 시 룸 단위 분할 탐색을 도입한다.

## 4. 런타임 데드락 감지

게임 중 매 "비가역 행동"(기폭, 돌을 데드스퀘어로 밀기) 직후, 현재 상태를 초기 상태로 한 1단계 솔버를 축소 실행해 해법 존재 여부를 확인한다. 소멸 시: "이 길은 막힌 것 같아… 다시 해볼까?" 안내 + 재시작 버튼. 규칙 엔진 공유(원칙 1) 덕분에 추가 구현 비용이 거의 없다.

## 5. Claude Code 구현 지시 (P4에서 07 문서로 상세화, 여기서는 계약만)

1. `rules.js`는 순수 함수로만 작성한다 — `nextState(state, action) → state | null`. 렌더링·DOM 의존 금지. 이것이 런타임·솔버 공유의 전제다.
2. stages.json 로드 시 본 문서 §2 스키마 검증을 먼저 수행하고, 위반 시 어떤 스테이지의 어떤 필드인지 명시하고 중단한다.
3. 솔버는 게임보다 먼저 구현한다. 첫 PR = rules.js + solve.js + 튜토리얼 스테이지 1개 SOLVABLE 판정. 게임 화면은 그 다음이다.

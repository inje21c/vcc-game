# vcc-game 포트폴리오 운영 계획

## 프로젝트 정의

`vcc-game`은 2003년에 시작된 모바일 게임 창작의 기록을 2026년의 바이브코딩과 인공지능 개발 방식으로 다시 여는 포트폴리오 프로젝트다. 목표는 원작 자료를 보존하고, 현대 웹/모바일 환경에서 다시 플레이 가능한 버전으로 리메이크하며, 제작 과정과 결과물을 공개 가능한 형태로 남기는 것이다.

## 핵심 목표

- 원작 보존: 원본 소스, 리소스, 기획 자료, 스테이지 데이터를 가능한 한 원형에 가깝게 보관한다.
- 모던 리뉴얼: 원작의 감정과 규칙을 유지하되 2026년 모바일 화면, 사운드, 설치 경험, 접근성에 맞게 재구축한다.
- 플레이 가능한 포트폴리오: 첫 화면에서 프로젝트의 역사와 의도를 소개하고, 사용자가 게임을 선택해 바로 플레이할 수 있게 한다.
- 베타 운영: 소수 테스터가 오류와 개선 의견을 남길 수 있는 리포트 흐름을 만든다.
- 공개와 수익화: 후원, 광고 동의, 선택형 광고, 설치형 PWA를 단계적으로 도입한다.

## 사이트 경험

첫 진입 화면은 단순한 랜딩 페이지가 아니라 포트폴리오 허브여야 한다.

1. 프로젝트 소개
   - 2003년 모바일 게임 창업과 원작 개발 맥락
   - 23년 뒤 AI와 바이브코딩으로 복원/리메이크하는 현재의 의미
   - 원작 보존본과 모던 리뉴얼본을 함께 전시한다는 원칙

2. 게임 선택
   - 각 게임은 원작, 모던 버전, 개발 노트, 변경 기록으로 나뉜다.
   - 현재는 `도와줘 헤이다`를 첫 대표작으로 둔다.

3. 플레이와 설치
   - 브라우저 즉시 플레이를 기본으로 한다.
   - 모바일 사용자는 PWA 설치를 안내받는다.
   - 설치 안내는 강제하지 않고, 플레이 흐름을 방해하지 않는다.

4. 후원과 광고 동의
   - 후원은 프로젝트 보존과 개발 지속을 돕는 선택 행동으로 둔다.
   - 광고는 명확한 동의 뒤에만 노출한다.
   - 광고는 게임 시작 전, 챕터 사이, 결과 화면 등 흐름을 해치지 않는 지점에만 배치한다.

5. 베타 피드백
   - 닫힌 테스트 기간에는 GitHub Issues 또는 외부 폼을 리포트 보드로 사용한다.
   - 공개 후에는 사이트 안에서 오류 보고 버튼을 제공하고, 내부적으로 이슈 보드에 연결한다.

## 폴더 원칙

```text
vcc-game/
  docs/                         전체 운영/개발 기준
  common/                       여러 게임이 공유하는 코드와 도구
  templates/                    새 게임 시작 템플릿
  games/
    help-me-heyda/              원작 복원과 원본 자료
    help-me-heyda-modern/       2026 모던 리뉴얼 버전
```

각 게임은 원작과 모던 버전을 분리한다. 원작 폴더는 보존과 분석을 우선하고, 모던 폴더는 플레이 가능한 현대판 구현을 우선한다.

## 단계별 실행 계획

### 1단계. 닫힌 베타 기반 정리

- `도와줘 헤이다 2026`의 기본 플레이 오류를 잡는다.
- 베타 테스터용 오류 리포트 보드를 연다.
- GitHub 커밋 후 Vercel 자동 배포 URL을 베타 기준 URL로 사용한다.
- 리포트 항목은 기기, 브라우저, 화면, 재현 절차, 기대 결과, 실제 결과, 스크린샷으로 통일한다.
- 수정 기준은 치명 오류, 진행 불가, 조작 불편, 시각/사운드 품질 순서로 둔다.

### 2단계. 포트폴리오 허브 구축

- 루트 `index.html`을 프로젝트 허브로 발전시킨다.
- 게임 카드에서 원작 보기, 모던 버전 플레이, 개발 문서 보기로 이동하게 한다.
- 각 게임마다 개발 로그와 버전 기록을 남긴다.

### 3단계. PWA와 설치 경험

- 모던 게임은 모바일 PWA 설치를 지원한다.
- 오프라인 캐시, 아이콘, 시작 화면, 세로 화면 최적화를 검증한다.
- 설치 후에도 오류 리포트와 버전 표시가 가능해야 한다.

### 4단계. 후원과 광고 동의

- 후원 버튼을 먼저 도입하고 광고는 나중에 붙인다.
- 광고는 선택 동의 후 활성화한다.
- 광고가 있는 빌드와 없는 빌드를 설정으로 구분할 수 있게 한다.

### 5단계. 공개 운영

- 공개 버전은 안정 태그를 붙인다.
- 베타 기능은 별도 URL 또는 설정 플래그로 분리한다.
- 오류 리포트, 변경 기록, 알려진 문제를 함께 공개한다.

## Vercel 배포 운영

GitHub와 Vercel이 연결되어 있으므로 배포 흐름은 다음을 기본으로 한다.

- `main` 또는 운영 브랜치 push: Production 배포
- 기능 브랜치 또는 PR push: Preview 배포
- 베타 테스터 공유: Production URL 또는 지정 Preview URL
- 리포트 작성: 플레이한 Vercel URL을 필수로 기록

배포 후 최소 확인:

1. 타이틀 화면이 로드되는지 확인한다.
2. 메뉴에서 스토리 모드로 진입한다.
3. Stage 1이 시작되는지 확인한다.
4. 서바이벌 모드로 진입한다.
5. 오류 보고 링크가 열리는지 확인한다.

## 운영 원칙

- 원작의 정체성을 훼손하지 않는 선에서 현대화한다.
- 개발 과정 자체도 포트폴리오의 일부로 기록한다.
- 광고와 후원은 플레이어의 신뢰를 해치지 않는 방식으로만 넣는다.
- 베타 리포트는 빠르게 받되, 수정 우선순위는 플레이 가능성에 둔다.
- 새 기능보다 현재 버전의 안정성을 먼저 확보한다.

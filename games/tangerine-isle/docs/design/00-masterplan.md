# 프로젝트 마스터플랜 — 2003 MUSTI 모던화 (가칭: 감귤섬 삼총사 / Tangerine Isle)

**문서 버전**: 0.1 (마스터플랜 — 상세 설계 이전 단계)
**작성일**: 2026-07-08
**선행 프로젝트**: 도와줘 헤이다 2026 (vcc-game) — 본 프로젝트는 그 패턴의 2회차 적용이자 "바이브코딩 게임 리메이크" 스킬화의 검증 케이스
**저장 지시**: 이 문서는 신규 레포지토리의 `docs/design/00-masterplan.md`에 그대로 저장한다. 이후 모든 상세 문서는 이 문서의 섹션 번호를 상위 참조로 인용한다.

## 0. 이 프로젝트가 무엇인가 — 세 겹의 목적

이 프로젝트는 단일 게임 리메이크가 아니라 세 층위의 자산을 동시에 만든다. 이 구분이 이후 모든 의사결정의 기준이다.

| 층위 | 산출물 | 성격 |
|---|---|---|
| L1: 게임 | 2003년 MUSTI의 모던 웹/PWA 리메이크 | 아카이브 + 포트폴리오 (헤이다와 동일 프레임) |
| L2: 파이프라인 | 설계문서 → 에셋 프롬프트 → Claude Code 구현의 재현 가능한 절차 | 헤이다에서 1회 수행한 것의 2회차 — 반복되면 패턴, 패턴이면 스킬화 가능 |
| L3: 스킬/에이전트 | `retro-game-remake` 스킬 (Claude Code용) | L2를 문서화·템플릿화한 메타 자산. VCC 커리큘럼/워크샵 교재와도 연결 |

**원칙**: L1을 만들면서 L2를 기록하고, L2가 두 번 검증된 시점(본 프로젝트 완료)에 L3를 추출한다. L3를 먼저 만들지 않는다 — 1회 사례로 만든 스킬은 과적합이다.

## 1. IP 구조 — 무엇을 버리고 무엇을 계승하는가

### 1.1 리스크의 정확한 위치

2003년 당시 MUSTI 캐릭터는 라이선스 계약 하에 사용했고, 현재는 그 계약이 없다. 따라서 리스크는 게임 메커니즘이 아니라 **캐릭터 IP와 게임명**에 국한된다. 게임 메커니즘(밀기 퍼즐 규칙 자체)은 저작권 보호 대상이 아니며, 당시 자사가 개발한 코드/레벨 설계의 권리는 별도 사안이다.

### 1.2 제거·대체 원칙

| 항목 | 처리 |
|---|---|
| MUSTI 캐릭터 (흰 고양이) | 외형 재설계. "흰 고양이"라는 속성은 유지하되 실루엣·비율·표정·복장을 원작과 식별 불가능하게 변경. 원작 스프라이트를 참조 이미지로 AI에 넣지 않는다 (스타일 전이 리스크) |
| 게임명 "MUSTI" | 완전 교체 (§2) |
| 브라운 토끼 / 초록 거북이 | 원작 라이선스 캐릭터인지 자체 제작인지 확인 필요. 자체 제작이었다면 계승 가능하나, 어차피 3종 모두 신규 아트로 통일하는 것이 일관성·안전성 모두 우위 |
| 레벨 구조·규칙 | 계승. 이것이 리메이크의 본체 |
| README/스토어 표기 | "2003년 자사 개발 모바일 퍼즐게임의 메커니즘을 계승한 리메이크"로 서술. 원작명·원 캐릭터명 비표기 |

### 1.3 확인 필요 (사용자 답변 대기)

- [ ] 토끼·거북이도 라이선스 캐릭터였는가, 자체 제작이었는가?
- [ ] 당시 게임 코드·레벨 데이터의 권리가 현재 본인/전 회사 중 어디에 있는가? (레벨을 그대로 복원할지, 참조만 할지의 갈림길)

## 2. 게임명 — 후보와 추천

### 2.1 명명 기준

① 상표 충돌 가능성 낮음 ② 검색 가능성(고유성) ③ 캐릭터 3종(고양이·토끼·거북이) 프레임 수용 ④ 시리즈 확장성(헤이다와 같은 "MobileAge 리마스터 시리즈" 우산 아래) ⑤ 서사적 뿌리.

### 2.2 후보

| 후보 | 영문 | 논리 |
|---|---|---|
| **감귤섬 삼총사** ★추천 | Tangerine Isle | 원작의 주황 과일을 감귤로 재해석 → 제주 세계관. 개인 서사(제주 출신, 검은모래 이야기)와 연결되어 IP가 "대체품"이 아니라 "고유 정체성"이 된다. LinkedIn 콘텐츠 서사로도 강력: "라이선스가 끝난 캐릭터를 고향의 이야기로 바꿨다" |
| 세돌이 원정대 | Three Pebbles | 바위 밀기 메커니즘 중심 명명. 무난하나 서사 없음 |
| 폴짝섬 | Hop Isle | 가볍고 캐주얼. 토끼 중심으로 기울어 3캐릭터 균형 약함 |
| 미루미루 | Mirumiru | "밀다"의 의성어화. 고유성 높으나 의미 전달력 낮음 |
| 조약돌 삼형제 | Pebble Trio | 안전하나 평범 |

### 2.3 추천 근거 — 감귤섬 삼총사

IP 교체를 비용이 아니라 자산으로 전환하는 유일한 후보다. 흰 고양이·갈색 토끼·초록 거북이가 감귤을 지키는 섬의 삼총사가 되는 순간, 캐릭터는 원작의 그림자가 아니라 독자적 세계의 주인이 된다. 슬라임은 감귤을 노리는 침입자로, 물 타일은 섬의 지형으로 자연스럽게 재해석된다. 검은모래 이야기(제주 가족 서사)와 세계관을 느슨하게 공유할 수도 있다 — 결합은 선택 사항으로 남긴다.

**최종 확정은 사용자가 한다.** 이후 문서에서는 확정 전까지 코드네임 `tangerine-isle`을 사용한다.

## 3. 원작 분석 — 관측 사실과 검증 대기 가설

### 3.1 확정 (영상·스크린샷 관측)

- 15×15 타일 그리드 (240×240px, 16px 타일), 고정 화면 스테이지형 퍼즐
- 타일: 모래 바닥 / 수풀(벽) / 물(애니메이션) / 바위 / 과일 / 출구 표지판
- 개체: 파란 슬라임 (인접 숫자 카운터 동반, 예: 8, 15)
- 플레이어 3종: 흰 고양이, 갈색 토끼, 초록 거북이 — 스테이지별 교체 등장
- 상태 화면: 게임오버(해골), 스테이지 결과
- 조작: 4방향 이동 (2003 피처폰 키패드 전제)

### 3.2 가설 — 원작 개발자 검증 필요 (GDD 착수의 전제조건)

| # | 가설 | 참이면 | 거짓이면 |
|---|---|---|---|
| H1 | 바위를 물에 밀어 넣어 다리를 만든다 | 소코반+지형변형 하이브리드 | 순수 소코반 변형 |
| H2 | 바위로 슬라임을 깔아 처치한다 | 전투가 퍼즐에 내장 | 슬라임은 회피 대상 |
| H3 | 슬라임 옆 숫자 = 카운트다운 (0이 되면 부화/이동 개시) | 시간압박이 턴 단위로 존재 | 숫자는 HP 또는 점수 |
| H4 | 클리어 = 과일 전량 수집 + 출구 도달 | 2중 목표 구조 | 단일 목표 |
| H5 | 캐릭터별 능력 차이 (예: 거북이 수영, 토끼 점프) | 캐릭터가 레벨 설계 변수 | 캐릭터는 스킨 |
| H6 | 이동은 턴제 (플레이어 1보 = 슬라임 카운터 1 감소) | 결정론적 퍼즐 → 레벨 검증기 제작 가능 | 리얼타임 요소 존재 |

H6이 특히 중요하다. 턴제 결정론이면 **레벨 솔버(자동 검증기)**를 만들 수 있고, 이것이 레벨 디자인 생산성과 L3 스킬의 핵심 컴포넌트가 된다.

### 3.3 총 스테이지 수, 아이템 종류(영상 중 열쇠/삽 추정 개체), 목숨 시스템 — 사용자 확인 대기

## 4. 개발 문서 체계 — 헤이다 패턴 계승 + 개선

상세화 단계에서 아래 문서를 순서대로 생산한다. 각각이 독립 세션 1회 분량이다.

```
docs/design/
├── 00-masterplan.md          ← 본 문서
├── 01-gdd.md                 게임 디자인 문서 (규칙 명세 — §3 가설 검증 후 착수)
├── 02-level-format.md        stages.json 스키마 + 레벨 검증기(솔버) 명세
├── 03-level-design.md        전체 스테이지 설계 (난이도 곡선, 챕터 구조)
├── 04-asset-spec.md          에셋 목록 전수 (스프라이트/타일/UI/사운드 ID 체계)
├── 05-prompt-pack-image.md   ChatGPT(이미지 생성) 프롬프트 팩 — 04의 ID와 1:1 매핑
├── 06-prompt-pack-audio.md   Suno 프롬프트 팩 (BGM 챕터별 + SFX 리스트)
└── 07-implementation.md      Claude Code 구현 지시서 (아키텍처, 파일 구조, PR 단위)
```

**헤이다 대비 개선점 두 가지:**

1. **에셋 ID 체계 선행** (04 → 05/06): 헤이다에서는 프롬프트와 에셋 파일명의 매핑이 사후적이었다. 이번에는 04에서 `spr_cat_idle_01` 식의 ID를 먼저 확정하고, 이미지/사운드 프롬프트가 그 ID를 목표 파일명으로 명시한다. 생성 → 리네이밍 → 배치의 수작업이 사라진다.
2. **레벨 솔버 내장** (02): H6이 참일 경우, 레벨 JSON을 입력하면 최단 해법과 클리어 가능 여부를 출력하는 검증기를 게임 코드보다 먼저 만든다. "풀 수 있는 레벨"이 설계 단계에서 보장된다.

## 7. L3 — `retro-game-remake` 스킬 설계 방향 (P5에서 상세화)

스킬의 본질은 "레트로 게임 영상/스크린샷 → 플레이 가능한 모던 웹 게임"의 절차 표준화다. 구조 스케치:

```
retro-game-remake/
├── SKILL.md                  트리거 조건, 6단계 절차 (본 로드맵의 일반화)
├── templates/
│   ├── masterplan.md         본 문서의 템플릿화 (IP 체크리스트 포함)
│   ├── gdd.md                규칙 명세 템플릿 (관측/가설 분리 프레임)
│   ├── asset-spec.md         ID 체계 규약
│   ├── prompt-pack-image.md  스프라이트/타일/배경 프롬프트 골격
│   └── prompt-pack-audio.md  Suno 프롬프트 골격 (기존 suno-music-workflow 스킬 참조)
└── scripts/
    └── level-solver-scaffold.js   턴제 퍼즐용 범용 솔버 골격
```

핵심 설계 판단: 스킬의 첫 단계를 항상 **"관측 사실과 가설을 분리하고, 가설을 원작 소유자에게 검증받는다"**로 고정한다. AI가 영상만 보고 규칙을 확신하는 것이 이 파이프라인의 최대 실패 지점이며, 이는 감리 방법론의 "증거와 추정의 분리" 원칙과 정확히 같은 구조다 — AI 감리 포지셔닝 서사와도 연결 가능한 지점.

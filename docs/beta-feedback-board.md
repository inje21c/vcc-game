# 베타 오류 리포트 보드 계획

## 목적

닫힌 베타 테스터가 `도와줘 헤이다 2026`을 플레이하면서 발견한 오류, 불편, 개선 의견을 한곳에 남기도록 한다. 지금 당장은 무거운 회원제 게시판보다 GitHub Issues 또는 간단한 폼 기반 보드가 적합하다.

## 1차 권장안: GitHub Issues

장점:

- 별도 서버와 데이터베이스가 필요 없다.
- 댓글, 상태 라벨, 첨부 이미지, 이력 관리가 된다.
- 코드 수정 커밋과 오류 리포트를 연결하기 쉽다.

운영 방식:

- 저장소에 Issues를 켠다.
- 베타 테스터에게 이슈 작성 링크를 공유한다.
- `.github/ISSUE_TEMPLATE/beta-game-report.yml` 템플릿을 사용한다.
- 라벨은 `beta`, `bug`, `feedback`, `help-me-heyda`, `mobile`, `pwa`를 기본으로 둔다.

## 베타 테스트 URL 기준

GitHub에 커밋하면 Vercel에 자동 배포되는 흐름을 베타 운영의 기본으로 둔다.

- Production URL: 현재 베타 기준 URL로 사용한다.
- Preview URL: 특정 브랜치나 PR의 기능을 먼저 검증할 때 사용한다.
- 테스터에게는 한 번에 하나의 기준 URL만 공유한다.
- 리포트에는 반드시 플레이한 URL을 남긴다.

베타 시작 조건:

1. GitHub에 변경사항을 커밋하고 push한다.
2. Vercel 배포가 성공했는지 확인한다.
3. 배포된 URL에서 타이틀, 메뉴, 스토리 Stage 1, 서바이벌 진입을 빠르게 확인한다.
4. 같은 URL을 테스터에게 공유한다.

운영 팁:

- 안정 테스트는 Production URL로 받는다.
- 새 기능 검증은 Preview URL로 먼저 받는다.
- 문제가 큰 Preview는 Production으로 올리지 않는다.
- Production에 문제가 생기면 직전 정상 배포로 rollback한다.

## 리포트 작성 기준

테스터는 아래 정보를 남긴다.

- 플레이한 게임과 버전
- 기기와 브라우저
- 발생 화면
- 재현 절차
- 기대한 결과
- 실제 결과
- 스크린샷 또는 화면 녹화
- 진행 불가 여부

## 우선순위 기준

- P0: 게임 시작 불가, 진행 불가, 저장 데이터 파손, 화면 전체 멈춤
- P1: 스테이지 클리어 불가, 조작 입력 오류, 결과 화면 오류
- P2: 특정 기기 UI 깨짐, 사운드 불편, 성능 저하
- P3: 문구, 밸런스, 연출, 편의성 개선

## 2차 적용안: 사이트 내 오류 보고 버튼

포트폴리오 허브와 게임 화면에 `오류 보고` 버튼을 넣는다. `도와줘 헤이다 2026`은 GitHub 로그인 없이 작성 가능한 `report.html` 폼을 사용하고, Vercel Function이 GitHub Issue를 대신 생성한다.

필드:

- 게임 ID
- 버전
- 현재 모드
- 현재 스테이지
- 기기 정보
- 브라우저 정보
- 사용자 설명
- 선택 첨부

가능하면 자동 수집:

- URL
- 화면 크기
- userAgent
- 최근 게임 상태
- 최근 콘솔 오류

필요한 Vercel 환경변수:

- `GITHUB_ISSUES_TOKEN`: GitHub Issues 생성 권한이 있는 fine-grained token 또는 classic token
- `BETA_REPORT_CODE`: 허가된 테스터에게만 공유하는 초대코드
- `GITHUB_REPOSITORY`: 기본값은 `inje21c/vcc-game`, 다른 저장소로 보낼 때만 설정

보안 기준:

- GitHub 토큰은 브라우저 코드에 넣지 않는다.
- 초대코드를 모르는 사용자는 API에서 거부한다.
- 텍스트 리포트만 먼저 받으며, 스크린샷 업로드는 추후 저장소 구조를 정한 뒤 추가한다.

## 3차 확장안: 자체 게시판

공개 운영 규모가 커지면 자체 게시판을 만든다.

- 로그인: GitHub, Google, 또는 이메일
- 저장소: Postgres 계열 DB
- 파일 첨부: 오브젝트 스토리지
- 관리자 화면: 상태, 라벨, 담당자, 해결 버전
- 공개 범위: 비공개 베타와 공개 제보를 분리

## 당장 해야 할 일

1. GitHub Issues를 베타 보드로 사용한다.
2. 베타 리포트 템플릿을 저장소에 추가한다.
3. Vercel Production URL을 현재 베타 기준 URL로 정한다.
4. `도와줘 헤이다 2026` 화면 또는 README에 리포트 링크를 노출한다.
5. 리포트가 들어오면 P0/P1을 먼저 처리한다.
6. 반복되는 리포트는 `known issues` 문서로 정리한다.
